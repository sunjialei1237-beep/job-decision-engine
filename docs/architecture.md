# 完整日流水线：编排模式参考

本仓库只开源了**决策与编排核心**（`/job-analyze` + `/job-filter`）。作者本地还跑一条更完整的「每日流水线」，把抓取、预筛、投递、反馈回填串成闭环。这条流水线里**抓取层和反馈同步层属于私有实现**（涉及目标平台合规与个人数据，不发布），但其**编排模式**值得参考——你完全可以照这个骨架接自己的数据源。

> ⚠️ 合规提示：抓取和自动写操作（自动投递/自动打招呼）在多数招聘平台的服务条款里是被禁止的。下面描述的是**手动投递 + 自动分析**的模式，刻意把写操作留给人。

## 日流水线骨架

```
[环境检查] → [抓取岗位] → [规则预筛] → [/job-filter 批量评分+话术] → [写投递清单]
                                                                          │
                                                                          ▼
                                                              [人工照清单【手动】投递]
                                                                          │
                                                                          ▼
                                                              [回填投递结果/HR回复]
                                                                          │
                                                                          ▼
                                                              [下一轮预筛剔除已投]
```

### 1. 环境检查（可选）
跑流水线前确认依赖就绪：数据源可达、登录态有效、`profile.md` 存在。任一缺失就停，别带着异常硬跑。

### 2. 抓取 → 预筛（你的数据源侧）
抓取岗位详情，用规则脚本打 `rule_score`、剔除已投/回避项、标 `is_new`，产出 `candidates.json`（schema 见 `data-schema.md`）。这两步在 LLM 之前，零 token 成本，是控成本的第一道闸门。

### 3. LLM 批量评分 + 话术（= `/job-filter`）
本仓库直接提供的部分。读 `candidates.json` → 并行评分 → top 40 → 并行话术 → 写投递清单。

### 4. 投递清单
`output/reports/<today>/投递清单.{md,csv}`。md 给人看（排名总览 + 逐家详情 + 可复制话术），csv 给 Excel/记账工具消费。

### 5. 人工投递（关键：不自动投递）
从 S 级开始，照话术在目标平台**逐家手动**发送。为什么不自动投递？写操作风控高、容易被封号，且群发式自动投递对平台和其他用户都是干扰。把分析自动化、把投递留给人，是性价比和合规的平衡点。

### 6. 反馈闭环（你的记账侧）
- 投完标「已投」；下架/不投标「未投」。
- 后续几天回填 HR 结果（回复/进面/offer/拒/无回复）。
- 下一轮预筛读这份账本，剔除已投、超期无回复的标掉，只推新的。

这一步让你能算出**真实的漏斗转化率**（触达 → 回复 → 面试 → offer），反过来校准画像和预筛规则。

## 复用这套模式

- **抓取/预筛/记账**：接你自己的实现，只要产出/消费上面约定的 JSON 即可。
- **评分/话术**：直接用本仓库的 `/job-filter` + 两个 agent。
- **闭环**：照第 6 步维护一份投递账本，定期回流校准。

把「分析」自动化、「数据获取与投递」留给合规的人/工具，是这个项目想表达的核心取舍。
